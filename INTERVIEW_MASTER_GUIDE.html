1. Python (Core & Async)

Q1: Explain the difference between synchronous and asynchronous Python. Why is your backend fully async?
🔑 TCS Keywords: Global Interpreter Lock (GIL), asyncio, Event Loop, I/O bound vs CPU bound, non-blocking
📖 Answer:

Python is traditionally synchronous and single-threaded due to the Global Interpreter Lock (GIL), meaning it executes one line of code at a time. If line 2 makes a network request to MongoDB or OpenAI, the entire server freezes and waits for the response (blocking I/O). In a web server, this means no other users can be served during that wait.

Asynchronous Python (`asyncio`) introduces an Event Loop. When we use the `await` keyword in front of an I/O operation, we are telling Python: "This operation will take some time over the network. Pause this function, yield control back to the Event Loop, and go serve other users. Wake this function up when the network response arrives." Our Adaptive RAG heavily relies on external APIs (OpenAI, Tavily) and Database calls (MongoDB). Making it async ensures our FastAPI server remains highly concurrent and can handle thousands of simultaneous queries without freezing.

Code Snippet from Project
@router.post("/rag/query")
async def rag_query(req: QueryRequest):
    chat_history = ChatHistory.get_session_history(req.session_id)
    await chat_history.add_message(HumanMessage(content=req.query))
    messages = await chat_history.get_messages()
    # The await keyword yields control back to FastAPI's event loop
❓ Why This Over Alternatives?

We chose async/await with FastAPI over synchronous threads (like Flask uses by default) because threads have high memory overhead and context-switching costs. Async handles I/O-bound tasks much more efficiently on a single thread.

src/api/routes.py -> rag_query() function.

Q2: How does Python's typing system work? Explain TypedDict and Annotated.
🔑 TCS Keywords: Type hints, static analysis, mypy, TypedDict, Annotated, Reducers
📖 Answer:

Python is dynamically typed, meaning variables can change types at runtime. However, for large projects, this causes runtime crashes. We use Python's `typing` module to add type hints, which act as documentation and allow IDEs to catch errors before execution. In LangGraph, strict typing is mandatory to define the "State" that passes between nodes.

We use `TypedDict` to define a dictionary where the keys are fixed and have specific types. This ensures our graph state always has exactly the keys we expect. We use `Annotated` to attach metadata to a type. Specifically, `Annotated[list[AnyMessage], add_messages]` tells LangGraph that the `messages` key is a list, but whenever a new node returns a message, it should use the `add_messages` function (a Reducer) to append it to the list, rather than overwriting the entire list.

Code Snippet from Project
from typing import TypedDict, Annotated, Optional
from langgraph.graph import add_messages

class State(TypedDict):
    messages: Annotated[nb_list[AnyMessage], add_messages]
    binary_score: Optional[str]
    route: Optional[str]
    latest_query: Optional[str]
❓ Why This Over Alternatives?

Why not a standard Python dict? A standard dict doesn't enforce key names or types, leading to KeyError exceptions at runtime. TypedDict enforces structure. Why not a Pydantic BaseModel here? LangGraph specifically requires TypedDict for its State schema to handle reducer metadata via Annotated.

📂 Practically Implemented In:

src/models/state.py -> State class.

Q3: Explain Context Managers in Python. How are they used for file handling?
🔑 TCS Keywords: with statement, __enter__, __exit__, resource leak, memory management
📖 Answer:

Context managers in Python are used to properly manage resources like file streams, database connections, and network sockets. They are implemented using the `with` statement, which inherently calls the `__enter__()` and `__exit__()` dunder methods of an object.

If you open a file using `f = open()`, you must explicitly call `f.close()`. If an exception occurs before `close()` is called, the file descriptor remains locked, leading to resource leaks. Using `with open()` guarantees that the file will be closed automatically the moment the block is exited, regardless of whether it exited normally or due to a crash.

Code Snippet from Project
def __init__(self, config_file: str = None):
    base_path = Path(__file__).parent
    config_path = base_path / "prompts.yaml" if config_file is None else Path(config_file)
    with open(config_path, "r") as f:
        self.config = yaml.safe_load(f)
    # File 'f' is automatically closed here
❓ Why This Over Alternatives?

Over try/finally blocks, the with statement is cleaner, more pythonic, and far less prone to developer error (forgetting the finally block).

📂 Practically Implemented In:

src/config/settings.py -> __init__ method.

2. FastAPI & Web Architecture

Q4: What is ASGI and how does FastAPI utilize it?
🔑 TCS Keywords: ASGI vs WSGI, Starlette, Uvicorn, asynchronous concurrency
📖 Answer:

WSGI (Web Server Gateway Interface) is the older Python standard used by frameworks like Flask and Django. It handles requests synchronously—one request blocks one thread. ASGI (Asynchronous Server Gateway Interface) is the modern standard that allows for asynchronous, non-blocking execution.

FastAPI is built on top of Starlette (an ASGI microframework) and runs on an ASGI server like Uvicorn. This means when FastAPI receives a request that requires waiting (like querying the OpenAI API), it doesn't block the thread. It suspends the request, serves other users, and resumes when the API responds. This allows our backend to handle massive throughput with very few system resources.

Code Snippet from Project
from fastapi import FastAPI
from src.api.routes import router

app = FastAPI(title="Adaptive RAG API")
app.include_router(router)
# This 'app' object is the ASGI callable that Uvicorn mounts
❓ Why This Over Alternatives?

We chose FastAPI over Flask because our RAG system is heavily I/O bound (waiting on OpenAI, Tavily, MongoDB, and FAISS). Flask would require complex worker setups (Gunicorn with Gevent) to handle high load, whereas FastAPI handles it natively via ASGI.

Q5: How do you handle file uploads in your FastAPI backend?
🔑 TCS Keywords: UploadFile, SpooledTemporaryFile, multipart/form-data, memory limits
📖 Answer:

When a user uploads a PDF from Streamlit, it is sent to FastAPI as `multipart/form-data`. In FastAPI, we use the `UploadFile` class rather than `bytes`. This is a critical architectural decision.

If we used `bytes`, the entire PDF would be loaded into the server's RAM. If 100 users upload 50MB PDFs, the server crashes from OOM (Out of Memory). `UploadFile` uses a `SpooledTemporaryFile`. It stores data in memory up to a certain size limit (default 1MB), and automatically writes anything larger directly to a temporary disk file. We then read this secure temporary file to process the LangChain document loaders.

Code Snippet from Project
@router.post("/rag/documents/upload")
async def upload_file(
    file: UploadFile = File(...), 
    description: str = Header(..., alias="X-Description")
):
    # UploadFile prevents memory exhaustion
    status_upload = documents(description, file)
    return {"status": status_upload}
❓ Why This Over Alternatives?

Using UploadFile over bytes ensures server stability. We also chose to pass the document description via a custom HTTP Header (X-Description) because mixing JSON bodies with multipart file uploads in HTTP is complex and error-prone.

📂 Practically Implemented In:

src/api/routes.py and processed in src/rag/document_upload.py.

3. LangChain & LangGraph Orchestration

Q6: What is LangGraph and how does it differ from a standard LangChain chain?
🔑 TCS Keywords: State Machine, Cyclic Graphs, Directed Acyclic Graph (DAG), deterministic execution
📖 Answer:

Standard LangChain sequences (using LCEL) are linear (A -> B -> C). If step B fails or needs to loop back to A, standard LangChain struggles. LangGraph solves this by treating the workflow as a Stateful Graph (State Machine).

In our project, we have a complex routing mechanism. The query goes to `query_analysis`. Depending on the LLM's classification, the graph branches to `retriever`, `general_llm`, or `web_search`. Furthermore, after `retriever`, it goes to `grade`. If the grade is "no" (irrelevant documents), the graph loops backward to `rewrite` the query and searches again. This cyclic, conditional logic is impossible with linear chains but trivial with LangGraph's nodes and edges.

Code Snippet from Project
graph = StateGraph(State)
graph.add_node("query_analysis", query_classifier)
graph.add_node("retriever", retriever_node)
graph.add_node("grade", grade)

# Conditional Routing based on classification
graph.add_conditional_edges("query_analysis", routing_tool)

# Standard edge
graph.add_edge("retriever", "grade")
❓ Why This Over Alternatives?

We chose LangGraph over standard LangChain Agents or AutoGPT because LangGraph provides exact, deterministic control over the execution flow. Standard agents are autonomous black boxes that can get stuck in infinite loops. LangGraph allows us to strictly define the fallback loops (like retrieving -> grading -> rewriting).

src/rag/graph_builder.py -> Bottom of the file.

Q7: Explain LCEL (LangChain Expression Language) and the Pipe (|) operator.
🔑 TCS Keywords: Composability, __or__ dunder method, Runnable protocol, streaming
📖 Answer:

LCEL is a declarative way to compose LangChain components. In Python, the `|` symbol is normally the bitwise OR operator. LangChain overrides the Python `__or__` dunder method on its base classes (Runnables) to create a data pipeline, similar to Unix pipes.

When we write `prompt | llm`, we are creating a new Runnable object. When invoked, it automatically passes the dictionary variables into the PromptTemplate to format the string, passes that string to the LLM, and returns the output. It handles all async wrapping and type conversions under the hood.

Code Snippet from Project
def rewrite_query(state: State):
    query = state["latest_query"]
    rewrite_prompt = PromptTemplate(
        template=config.prompt("rewrite_prompt"),
        input_variables=["query"]
    )
    
    # LCEL Pipe Operator
    chain = rewrite_prompt | llm
    
    result = chain.invoke({"query": query})
    return {"latest_query": result.content}
❓ Why This Over Alternatives?

Without LCEL, we would have to write imperative code: formatted_text = prompt.format(query=query) then response = llm.invoke(formatted_text). LCEL condenses this, makes it easily swap components, and natively supports parallel execution and streaming.

src/rag/graph_builder.py -> rewrite_query().

4. Vector Databases & FAISS

Q8: How does Vector Similarity Search actually work mathematically?
🔑 TCS Keywords: Embeddings, High-dimensional space, Cosine Similarity, Dot Product, Dense vectors
📖 Answer:

When we upload a document, we pass the text through an embedding model (like OpenAI's `text-embedding-3`). This model converts words into a dense vector—an array of floating-point numbers (e.g., 1536 dimensions). This array represents the semantic meaning of the text in high-dimensional space.

When a user asks a question, we embed the question into the exact same vector space. To find the answer, we don't do keyword matching. Instead, we use mathematical formulas—most commonly Cosine Similarity. Cosine similarity calculates the cosine of the angle between two vectors. If the angle is 0 (cosine = 1), the texts are identical in meaning. If the angle is 90 degrees (cosine = 0), they are unrelated. FAISS rapidly calculates the distance between the query vector and all document vectors, returning the closest ones (k-Nearest Neighbors).

Code Snippet from Project
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS

embeddings = OpenAIEmbeddings() # Converts text to 1536-dim vectors

def retriever_chain(chunks: list[Document]):
    global _faiss_vectorstore
    # FAISS mathematically stores and indexes these embeddings
    vectorstore = FAISS.from_documents(
        documents=chunks,
        embedding=embeddings
    )
    _faiss_vectorstore = vectorstore
❓ Why This Over Alternatives?

Traditional SQL databases use inverted indexes (like BM25 or Elasticsearch) for keyword matching. If the user asks "How do I fix a broken pipe?", SQL won't match a document about "Plumbing repairs" because the exact words don't match. Vector search understands that the semantic meaning is identical.

Q9: Explain HNSW (Hierarchical Navigable Small World) in FAISS.
🔑 TCS Keywords: Facebook AI Similarity Search, KNN vs ANN, indexing algorithms, O(log N)
📖 Answer:

FAISS (Facebook AI Similarity Search) is the underlying engine we use to store embeddings. If we have millions of document chunks, comparing the query vector against every single chunk (Exact KNN / Flat Search) requires O(N) time complexity, which is incredibly slow.

Instead, FAISS uses ANN (Approximate Nearest Neighbors) algorithms, specifically HNSW (Hierarchical Navigable Small World). HNSW builds a multi-layered graph. The top layer has very few points connected by long edges (like interstates). The bottom layers contain all points densely connected (like local streets). When searching, HNSW starts at the top layer, takes giant leaps towards the general direction of the query vector, drops down a layer, fine-tunes the position, and repeats until it reaches the bottom layer. This drops the search time complexity to O(log N), making vector retrieval instantaneous.

❓ Why FAISS Over Qdrant/Pinecone?

While the project config variables mention Qdrant, the actual implementation uses FAISS. I chose FAISS because it runs completely in-memory within the Python process. It requires no external Docker containers or API calls (unlike Pinecone), making it zero-latency and perfectly portable for this specific application scope.

5. OpenAI & ReAct Agents

Q10: What is the ReAct Framework and how is it used in your agent?
🔑 TCS Keywords: Reasoning and Acting, create_react_agent, AgentExecutor, intermediate_steps
📖 Answer:

ReAct stands for Reasoning and Acting. Standard LLMs just generate text blindly. An Agent has access to "Tools" (like a calculator, web browser, or our FAISS database) and uses a specific prompt structure to interact with them.

The loop works like this:
1. Thought: The LLM thinks about what it needs to do ("I need to find the user's uploaded document about policies").
2. Action: It decides to use the `retriever_customer_uploaded_documents` tool.
3. Observation: The tool runs and returns the text from FAISS back to the LLM.
4. Final Answer: The LLM formulates the answer. If the observation wasn't enough, it loops back to Thought and searches again.

Code Snippet from Project
from langchain.agents import create_react_agent, AgentExecutor

# The tool is our FAISS retriever
tools = [get_retriever()] 

react_agent = create_react_agent(llm, tools, prompt)

# The executor handles the looping logic
agent_executor = AgentExecutor(
    agent=react_agent,
    tools=tools,
    handle_parsing_errors=True,
    max_iterations=2, # Prevents infinite looping
    return_intermediate_steps=True
)
❓ Why This Over Alternatives?

Instead of manually querying FAISS, passing it to an agent allows the LLM to decide exactly what keywords to search for. If the first search yields bad results, the ReAct loop allows the agent to modify its search terms and try again autonomously up to 2 times.

6. Pydantic & Structured Outputs

Q11: How do you guarantee the LLM returns the exact format you need for routing?
🔑 TCS Keywords: Pydantic BaseModel, with_structured_output, Function Calling, schema validation
📖 Answer:

LLMs are non-deterministic text generators. If I need the LLM to act as a router and output "index" so my Python `if` statement works, the LLM might output "I think the best route is index." This causes a string mismatch and the app crashes.

To solve this, I use Pydantic models combined with OpenAI's Function Calling API via `with_structured_output`. Pydantic is a data validation library. By defining a Pydantic `BaseModel` called `RouteIdentifier`, LangChain converts this Python class into a JSON Schema. It passes this schema to the OpenAI API, forcing the model to output a strict JSON object that perfectly matches my class. When it returns, Pydantic parses the JSON back into a Python object, guaranteeing type safety.

Code Snippet from Project
from pydantic import BaseModel

class RouteIdentifier(BaseModel):
    route: str

# In graph_builder.py:
llm_with_structured_output = llm.with_structured_output(RouteIdentifier)
chain = classify_prompt | llm_with_structured_output

result = chain.invoke({"question": question, "context": context})
# result is guaranteed to be a RouteIdentifier object, not a string
return result.route

src/models/route_identifier.py and src/rag/graph_builder.py.

7. MongoDB & Motor

Q12: Why MongoDB? And why use the 'Motor' driver instead of PyMongo?
🔑 TCS Keywords: NoSQL, unstructured data, AsyncIOMotorClient, event loop blocking
📖 Answer:

Chat history is inherently semi-structured. Some messages are from humans, some from AI, some contain complex nested metadata (like `additional_kwargs` for tool calls). Using a SQL database like PostgreSQL would require complex JSONB columns or multiple normalized tables. MongoDB, as a NoSQL document database, natively stores data as BSON (Binary JSON), making it perfectly suited for storing unstructured LangChain message objects.

As for the driver, standard PyMongo is synchronous. If we used PyMongo in our async FastAPI app, every database read/write would block the event loop, destroying our concurrency. I used Motor (`AsyncIOMotorClient`), which is the official asynchronous driver for MongoDB. Motor leverages Python's `asyncio` to allow the server to handle other web requests while the database operates in the background.

Code Snippet from Project
from motor.motor_asyncio import AsyncIOMotorClient

MONGO_URL = "mongodb://localhost:27017"
client = AsyncIOMotorClient(MONGO_URL)
collection = client["adaptive_rag"]["chat_history"]

# Example async insertion:
async def add_message(self, message: BaseMessage) -> None:
    await collection.insert_one({
        "session_id": self.session_id,
        "content": message.content
    })

src/db/mongo_client.py and src/memory/chat_history_mongo.py.

8. Streamlit Frontend Architecture

Q13: How does Streamlit handle state, and why is `st.session_state` necessary?
🔑 TCS Keywords: Statelessness, re-rendering, session_state, state persistence
📖 Answer:

Streamlit is designed to completely rerun the entire Python script from top to bottom every time an interaction occurs (like a button click or text input). Because of this top-down re-execution, standard Python variables are reset and lost instantly.

To preserve data across these reruns, we must use `st.session_state`. In my chat interface, I store the JWT authentication token, the unique `session_id`, and the `chat_history` list inside `st.session_state`. When the user types a new message and the script reruns, I append the message to `st.session_state.chat_history` and then iterate over it to draw the entire conversation on the screen.

Code Snippet from Project
# Initialize chat history if it doesn't exist
if "chat_history" not in st.session_state:
    st.session_state.chat_history = []

user_input = st.chat_input("Ask a question...")

if user_input:
    # Store in session state so it survives the rerun
    st.session_state.chat_history.append(("user", user_input))
    response = query_backend(user_input, st.session_state["jwt_token"])
    st.session_state.chat_history.append(("assistant", response))
    st.rerun()  # Force UI update

streamlit_app/pages/chat.py.

9. End-to-End Walkthrough

The Complete Pipeline Trace
Explain this exactly as written if asked "Walk me through the architecture."
  1. User Input (Frontend): The user types a question in the Streamlit UI. Streamlit saves this to session_state and makes an HTTP POST request to our FastAPI backend using the requests library.
  2. API Reception (Backend): FastAPI receives the JSON payload via the /rag/query route. Pydantic validates the incoming payload into a QueryRequest object.
  3. Memory Integration: The async MongoDB Motor client fetches all previous chat history for this user's session_id and appends the new question to the database context.
  4. LangGraph Orchestration (The Router): The context enters the LangGraph workflow at the query_analysis node. An LLM analyzes the query and strictly outputs a RouteIdentifier (e.g., "index", "general", or "search").
  5. Execution Path:
    • If General: Routes to a standard LLM to just talk to the user.
    • If Search: Routes to the Tavily API to scrape the live internet for recent news.
    • If Index: Routes to the ReAct agent, which searches our in-memory FAISS database using OpenAI Embeddings and Cosine Similarity to find relevant PDF chunks.
  6. Self-Correction Loop: After FAISS retrieval, the graph moves to a grade node. An LLM checks if the retrieved text actually answers the question. If yes, it generates the final answer. If no, a conditional edge loops back to a rewrite_query node to try searching again with better keywords.
  7. Response: The generated answer is saved to MongoDB as an AIMessage, and FastAPI returns the string back to Streamlit where it is rendered to the user.
