🧠 Adaptive RAG: Complete Project Overview & Parameters Guide

🔄 Full End-to-End Project Flow

This is the exact sequence of events that happens when the application runs.

1. Application Startup

FastAPI boots up using Uvicorn (an ASGI server). The OpenAI LLM (gpt-4o), OpenAI Embeddings, and the asynchronous MongoDB connection are initialized into memory.

2. Document Upload Phase

User uploads a PDF via Streamlit. Streamlit sends a POST request to FastAPI. FastAPI temporarily saves the file, chunks the text, creates vector embeddings for each chunk, and stores them in the in-memory FAISS vector database.

3. User Query Phase

User asks a question. Streamlit sends the query to FastAPI. FastAPI saves the Human Message to MongoDB asynchronously. The query is then passed into the LangGraph Orchestrator.

4. LangGraph Routing (The "Adaptive" part)

The first node in LangGraph is the query_analysis node. The LLM reads the question and strictly outputs a route: "index" (search documents), "search" (search web), or "general" (normal chat).

5. Retrieval & ReAct Execution

If the route is "index", the graph moves to the retriever_node. Here, a ReAct Agent uses the FAISS retriever tool to autonomously search the database, adjusting its search terms until it finds the best context.

6. Grading & Rewriting Loop

The retrieved context goes to the grade node. An LLM grades it "yes" or "no". If "no" (context is irrelevant), the graph loops backward to a rewrite_query node to try searching again. If "yes", it moves forward.

7. Generation & Response

The context reaches the generate node, where the LLM writes the final answer. FastAPI saves this AI Message to MongoDB and returns it to Streamlit to display to the user.

🛠️ Tech Stack & Exact Parameters Deep Dive

Memorize these parameters. Interviewers love asking "What chunk size did you use?" or "How many iterations did you limit the agent to?" to prove you actually wrote the code.

1. FastAPI (Web Framework)

What is used: FastAPI and Uvicorn.
What is the use: Acts as the backend API layer to receive requests from Streamlit and run LangGraph.
Why it is used: Native support for asynchronous programming (async def), which is critical for preventing server freezes while waiting for OpenAI or MongoDB. It also provides automatic Pydantic validation.
Why ONLY this (Alternatives): We rejected Flask/Django because they are WSGI (synchronous by default) and handle high-latency I/O (like LLM API calls) poorly without complex multi-threading setups.
Parameter / CodeValue / DefinitionWhy this exact parameter?
file: UploadFile = File(...) FastAPI UploadFile class Used instead of raw bytes because UploadFile writes to a temporary disk file if the PDF is large, preventing the server from running out of RAM (OOM errors).
alias="X-Description" HTTP Header Binding Used in Header(..., alias="X-Description") to accept document descriptions via HTTP headers, keeping file uploads separate from JSON payloads.

2. RecursiveCharacterTextSplitter (Document Chunking)

What is used: LangChain's RecursiveCharacterTextSplitter.
What is the use: Breaks massive 100-page PDFs into small paragraphs before vectorization.
Why it is used: LLMs have strict context window limits. You cannot feed an entire book into an LLM. We must chunk it.
Why ONLY this (Alternatives): We rejected CharacterTextSplitter (which just splits by raw characters) because it chops words and sentences in half. The Recursive splitter tries to split by paragraphs (\n\n), then sentences (.), keeping semantic meaning intact.
Parameter / CodeValue / DefinitionWhy this exact parameter?
chunk_size 1000 (characters) 1000 characters (approx 200-250 words) is the sweet spot. Too large = too much noise for the LLM. Too small = not enough context to answer a question.
chunk_overlap 150 (characters) Critical parameter! If a sentence is split perfectly in the middle between Chunk 1 and Chunk 2, the meaning is lost. 150 overlap means the last 150 chars of Chunk 1 are duplicated at the start of Chunk 2, preserving context across boundaries.

3. FAISS (Vector Database)

What is used: Facebook AI Similarity Search (FAISS) via langchain_community.
What is the use: Stores the 1536-dimensional embeddings and performs rapid mathematical similarity search to find relevant text chunks.
Why it is used: It uses the HNSW (Hierarchical Navigable Small World) algorithm to search massive datasets in O(log N) time rather than comparing against every single document (O(N)).
Why ONLY this (Alternatives): While the config mentions Qdrant, the actual code uses FAISS. We chose FAISS over Qdrant, Pinecone, or PostgreSQL(pgvector) because FAISS runs 100% in-memory locally. It requires zero cloud database setup, zero network latency, and makes the project highly portable.
Parameter / CodeValue / DefinitionWhy this exact parameter?
embedding OpenAIEmbeddings() Passed into FAISS.from_documents(). Translates the text into OpenAI's dense vector space format so FAISS knows how to map the math.

4. ReAct Agent (AgentExecutor)

What is used: LangChain create_react_agent and AgentExecutor.
What is the use: Acts as an autonomous brain that decides *what* to search for in FAISS based on the user's query.
Why it is used: Standard RAG just takes the user's exact words and searches the DB. A ReAct agent reads the query, Thinks about what keywords to extract, Acts by calling the FAISS tool, and Observes the result.
Parameter / CodeValue / DefinitionWhy this exact parameter?
max_iterations 2 If the agent can't find the answer, it will keep retrying. Setting this to 2 prevents infinite loops and stops it from burning through OpenAI API credits.
handle_parsing_errors True If the LLM hallucinates an output format that breaks the agent's logic, this parameter catches the error and tells the LLM "You formatted that wrong, try again" instead of crashing the app.
return_intermediate_steps True Allows us to extract exactly which tool the agent decided to call and what input it used, which is saved to our state as tool_calls metadata for debugging.

5. OpenAI (LLM & Function Calling)

What is used: ChatOpenAI from langchain_openai.
What is the use: The brain of the entire system (Routing, Grading, Rewriting, Generating).
Why ONLY this: OpenAI has the most reliable "Function Calling" (Structured Outputs) capabilities on the market. Open-source models (like Llama) often fail to adhere strictly to JSON schemas, which would break our LangGraph routing logic.
Parameter / CodeValue / DefinitionWhy this exact parameter?
model "gpt-4o" The flagship model. Chosen because its reasoning capabilities are required to accurately "grade" document relevance and act as a reliable router.
.with_structured_output() RouteIdentifier (Pydantic) Forces the LLM to output a strict JSON object (e.g. {"route": "index"}) instead of conversational text. We pass our Pydantic classes (RouteIdentifier, Grade, VerificationResult) directly into this parameter.

6. MongoDB & Motor (Database)

What is used: AsyncIOMotorClient and a custom BaseChatMessageHistory class.
What is the use: Persists the conversation history across sessions.
Why it is used: Chat messages are JSON-like objects (HumanMessage, AIMessage, tool metadata). MongoDB is a document NoSQL database that stores data as BSON (Binary JSON), making it a perfect 1:1 mapping for our data.
Why ONLY Motor (Alternatives): We absolutely could NOT use the standard pymongo library because it is synchronous. If we used PyMongo, writing chat history to the DB would block FastAPI's event loop. motor.motor_asyncio is fully asynchronous.
Parameter / CodeValue / DefinitionWhy this exact parameter?
collection.find(...).sort() ("timestamp", 1) The 1 parameter in MongoDB sorting means "Ascending". This guarantees that when we pull chat history into the LLM context window, the oldest messages are first and the newest are last, maintaining timeline sanity.

7. LangGraph (Orchestrator)

What is used: StateGraph, Nodes, and Conditional Edges.
What is the use: Replaces linear code with a cyclical state machine.
Why ONLY LangGraph (Alternatives): Standard LangChain SequentialChain or LCEL cannot loop backwards. If our grader decides the documents are bad, we need to loop backward to the rewrite node. LangGraph is the only framework that supports deterministic cyclical graphs (DAGs with cycles).
Parameter / CodeValue / DefinitionWhy this exact parameter?
State (TypedDict) messages: Annotated[list, add_messages] We use Annotated to pass the add_messages reducer to LangGraph. This parameter is crucial: it tells the graph "When a node returns a new message, do NOT overwrite the existing message list. Append the new message to the list instead."
graph.add_conditional_edges ("query_analysis", routing_tool) This parameter links the exit of the analysis node to a python function (routing_tool). That function reads the state and dynamically returns the name of the next node to execute.
